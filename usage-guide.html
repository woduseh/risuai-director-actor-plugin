RisuAI Plugin V3 · Continuity Director · v0.8.0

RisuAI Continuity Director 사용 가이드

Continuity Director는 RisuAI의 메인 RP 모델 앞뒤에서 디렉터 패스를 수행하고, 장기 메모리를 정리하고, 현재 대화 흐름에 맞는 정보를 다시 불러와 장면의 연속성을 유지하는 플러그인입니다.

중요: 배포에 사용하는 실제 플러그인 파일은 dist/risuai-continuity-director-plugin.js입니다.
핵심 역할
내러티브 가이드

메인 RP 모델에 앞서 디렉터 브리프를 만들고 방향을 정리합니다.

메모리 구조
다층 메모리

Canonical Store, 메모리 문서, Session Notebook, Embedding 검색을 함께 씁니다.

운영 인터페이스
풀스크린 대시보드

설정, 프롬프트, 메모리, 프로필, 임베딩 상태를 한곳에서 관리합니다.

읽는 방법

처음 쓰는 경우에는 1~5번까지만 읽어도 바로 사용할 수 있습니다.

1. 빠른 시작

아래 순서대로 진행하면 가장 빠르게 기본 운용 상태에 도달할 수 있습니다.

1. 플러그인 불러오기

RisuAI Plugin V3에서 번들 파일을 가져옵니다.

dist/risuai-continuity-director-plugin.js

2. 대시보드 열기

플러그인 설정 또는 등록 버튼으로 풀스크린 대시보드를 엽니다.

3. 플러그인 활성화

General 탭에서 Enabled를 켭니다.

4. 디렉터 모델 연결

Model Settings에서 제공자에 맞는 인증 방식과 모델을 설정하고 연결 테스트를 합니다.

5. 현재 채팅에서 메모리 생성

Memory & Cache에서 Extract Current Chat 또는 Run Extract Now를 사용합니다.

6. 대화 시작

이제 Director 패스와 후처리 메모리 갱신이 자동으로 동작합니다.

2. 설치와 적용

이 플러그인은 단일 JS 번들로 배포됩니다. 사용자는 보통 빌드 결과물만 가져오면 됩니다.

권장 방식: 이미 빌드된 dist/risuai-continuity-director-plugin.js 파일을 RisuAI의 Plugin V3 가져오기 흐름으로 추가하세요.
  1. RisuAI에서 Plugin V3 가져오기 또는 로드 메뉴를 엽니다.
  2. dist/risuai-continuity-director-plugin.js 파일을 선택합니다.
  3. 플러그인이 등록되면 설정 화면 또는 버튼으로 대시보드를 엽니다.
  4. 디렉터 모델과 필요 시 임베딩 모델을 설정합니다.
  5. 사용할 캐릭터와 채팅을 연 상태에서 메모리 추출을 한 번 실행합니다.
주의: 이 플러그인은 현재 열려 있는 캐릭터/채팅을 기준으로 동작합니다. 메모리 작업을 실행하기 전에 반드시 대상 채팅이 맞는지 확인하세요.
제공자별 인증 방식: OpenAI, Anthropic, Google, Custom은 일반적으로 API 키 기반입니다. GitHub Copilot은 붙여넣은 Copilot token으로 직접 연결을 확인하고, Vertex AI는 서비스 계정 JSON 키를 붙여넣어 Gemini 호출과 임베딩 호출에 사용합니다.

3. 작동 방식 이해하기

Continuity Director는 단순한 설정 UI가 아니라, RP 진행 전후에 개입하는 메모리/연출 계층입니다.

Director 패스

사용자 입력이 들어오면 먼저 Director가 현재 상황과 메모리를 읽고 짧은 브리프를 만듭니다.

캐릭터 응답

메인 RP 모델은 두 가지 경로로 맥락을 전달받습니다: Director 브리프(짧은 연출 지시)와 별도의 장기 메모리 블록(CBS 메모리·Past Summary 인근에 삽입).

사후 검토

응답이 끝나면 후처리 패스가 대화 결과를 검토하고 메모리 후보를 정리합니다.

장기 메모리 유지

추출, 회상, 통합, 임베딩 갱신을 통해 장기적인 연속성을 관리합니다.

레이어 역할 언제 보나
Canonical Store 요약, 연속성 사실, 세계관 사실, 엔티티, 관계 같은 핵심 메모리의 기본 저장소 Memory & Cache 페이지의 편집 가능한 카드
메모리 문서 문서 단위로 쪼개진 장기 메모리 저장소 Memory Workbench의 읽기 전용 문서 목록
Session Notebook 현재 세션의 단기 흐름을 정리하는 노트 Memory Workbench 하단의 세션 노트
Embeddings 문서 후보를 의미 기반으로 미리 좁히는 검색 보조 계층 Embedding Status, Refresh Embeddings, 문서별 임베딩 배지

4. 대시보드 둘러보기

대시보드는 다섯 개의 메인 탭으로 구성됩니다. 처음에는 아래 흐름으로 익히면 편합니다.

탭 무엇을 하나 언제 쓰나
General 플러그인 활성화, 개입 강도, Director Mode, 삽입 방식, 메트릭 확인 초기 연결 직후와 기본 동작 점검 시
Prompt Tuning 브리프 소프트 캡, 사후 리뷰, 장기 메모리 주입, 임베딩 사용 여부, 프롬프트 프리셋 관리 개입 강도나 스타일을 조절할 때
Model Settings Director 모델과 임베딩 모델의 제공자, 인증 방식, 모델, 연결 테스트 제공자를 바꾸거나 오류를 점검할 때
Memory & Cache 핵심 메모리 편집, 추출/재생성/통합, 회상 점검, 임베딩 갱신, 메모리 문서 검사 연속성 관리의 중심 화면
Settings Profiles 설정 프로필 저장, 복제, 내보내기, 가져오기 캐릭터별 운용 스타일을 저장할 때

General 탭의 핵심

Enabled를 켜지 않으면 다른 설정이 있어도 Director 개입이 일어나지 않습니다.

Prompt Tuning의 핵심

Brief Token Soft Cap은 Director 브리프 출력에만 적용되는 소프트 캡입니다. 장기 메모리는 별도 경로로 주입되므로 이 값을 올려도 장기 메모리 크기에는 영향이 없습니다.

Model Settings의 핵심

Director 모델은 메인 RP 모델과 분리됩니다. 즉, RP 본체 모델을 바꾸지 않고도 디렉팅 품질을 조절할 수 있습니다.

Profiles의 핵심

프로필은 운용 방식 저장용입니다. 캐릭터별로 다른 성향을 빠르게 전환하려면 프로필을 적극적으로 쓰세요.

5. 메모리 & 캐시 페이지 사용법

이 페이지는 Continuity Director의 핵심입니다. 현재 열린 캐릭터/채팅 기준으로 메모리를 보고, 고치고, 다시 생성할 수 있습니다.

운영 원칙: 현재 페이지의 메모리 작업은 활성 캐릭터와 활성 채팅을 기준으로 수행된다고 생각하면 됩니다.

5-1. 편집 가능한 핵심 메모리

  • Summaries — 긴 흐름을 짧게 압축한 요약
  • Continuity Facts — 반드시 유지해야 하는 연속성 사실
  • World Facts — 세계관 설정과 배경 정보
  • Entities — 인물, 집단, 사물 등의 핵심 엔티티
  • Relations — 엔티티 간 관계

이 영역에서는 수동 추가/편집/삭제/선택 삭제가 가능합니다. 자동 추출 결과를 운영자가 직접 다듬는 용도입니다.

5-2. Memory Operations 카드

이 카드는 추출, 통합, 회상 점검, 임베딩 갱신 같은 운영 명령을 모아 둔 곳입니다. 특히 Extract와 Regenerate는 이름이 비슷해 보여도 역할이 다릅니다.

버튼 기능 권장 사용 시점
Run Extract Now 현재 상태를 기준으로 즉시 추출을 수행합니다. 새 채팅을 시작했거나 메모리가 비어 있을 때
Regenerate from Current Chat 현재 채팅을 기준으로 scoped 메모리를 다시 만드는 파괴적 재생성 작업입니다. 메모리 구조가 크게 꼬였거나 처음부터 다시 뽑고 싶을 때
Run Consolidation Now 중복 문서 통합과 정리를 강제로 실행합니다. 장기간 쌓인 메모리를 한 번 정리하고 싶을 때
View Recalled Docs 마지막 회상에 사용된 문서를 봅니다. 왜 특정 메모리가 사용되었는지 확인할 때
Toggle Fallback Retrieval 모델 기반 회상과 키워드 기반 회상을 전환합니다. 회상 품질을 비교하거나 문제를 진단할 때
Refresh Embeddings 현재 scope의 메모리 문서 임베딩을 다시 계산합니다. 임베딩 설정을 바꾼 뒤나 cache 상태가 stale일 때
Regenerate는 Extract와 다릅니다. Extract는 현재 흐름에서 메모리를 더 보강하는 쪽에 가깝고, Regenerate는 현재 채팅을 기준으로 메모리 저장소를 다시 만드는 더 강한 작업입니다.
파괴적 작업은 2단계 확인을 사용합니다. 삭제, 선택 삭제, 재생성 같은 작업은 첫 클릭에서 실행되지 않을 수 있습니다. 먼저 버튼이 armed 상태로 바뀌고, 다시 한 번 눌러야 실제로 수행됩니다.

5-3. Memory Workbench

Memory Workbench는 메모리 문서 저장소를 읽기 전용으로 보여 주는 검사 도구입니다.

  • Type / Status / Source 필터로 문서를 좁혀 볼 수 있습니다.
  • 문서별 Embedded / Not Embedded 상태를 확인할 수 있습니다.
  • MEMORY.md Preview로 현재 scope의 정리된 관점을 빠르게 볼 수 있습니다.
  • Session Notebook으로 현재 세션의 단기 흐름을 확인할 수 있습니다.
구분: Canonical 메모리 카드 영역은 편집용, Memory Workbench는 검사용입니다. 둘은 같은 목적이 아니라 서로 다른 시야를 제공합니다.

6. 추천 운용 흐름

새 캐릭터에 처음 적용할 때

  1. General에서 Enabled를 켭니다.
  2. Model Settings에서 Director 연결을 완료합니다.
  3. Memory & Cache에서 Extract Current Chat을 실행합니다.
  4. Summaries와 Continuity Facts를 빠르게 검수합니다.

긴 채팅이 누적되었을 때

  1. Run Extract Now로 최신 흐름을 추출합니다.
  2. Run Consolidation Now로 중복/파편 메모리를 정리합니다.
  3. Memory Workbench에서 stale 문서와 source를 훑어봅니다.

회상이 이상할 때

  1. View Recalled Docs로 어떤 문서가 뽑혔는지 확인합니다.
  2. 필요하면 Toggle Fallback Retrieval로 키워드 회상과 비교합니다.
  3. 임베딩 설정을 바꿨다면 Refresh Embeddings를 실행합니다.

캐릭터마다 다른 운용을 할 때

  1. Settings Profiles에서 프로필을 분리해 둡니다.
  2. 개입 강도, 프롬프트, 모델 조합을 캐릭터 성격에 맞게 저장합니다.
  3. 필요할 때 프로필을 불러와 적용합니다.

7. 주의사항과 팁

대시보드는 현재 채팅 기준이다

메모리 추출, 재생성, 통합, 임베딩 갱신은 모두 현재 보고 있는 채팅을 기준으로 실행한다고 생각하는 것이 안전합니다.

브리프와 장기 메모리는 별개다

Brief Token Soft Cap은 Director 브리프에만 적용되는 소프트 캡입니다. 장기 메모리는 CBS 메모리·Past Summary 인근에 별도로 삽입되어 RP 모델이 풍부한 맥락을 참고합니다.

프로필을 먼저 만들어 두면 편하다

캐릭터 성향이 자주 바뀌거나 실험을 많이 한다면 프로필을 먼저 분리해 두는 편이 관리가 쉽습니다.

임베딩은 보조 계층이다

임베딩 검색은 회상 후보를 좁히는 보조 장치입니다. 문제가 있으면 Fallback Retrieval로 안전하게 비교할 수 있습니다.

8. 문제 해결

증상 먼저 볼 것 권장 조치
Director가 개입하지 않는 것 같다 General 탭의 Enabled, Model Settings의 연결 상태 플러그인을 켜고, Director 모델 연결 테스트를 다시 실행합니다.
메모리가 전혀 보이지 않는다 Memory & Cache 페이지, Extract Current Chat / Run Extract Now 현재 채팅을 연 상태에서 추출을 한 번 실행합니다.
회상이 이상하거나 부정확하다 View Recalled Docs, Toggle Fallback Retrieval, Embedding Status 마지막 회상 문서를 확인하고, 필요하면 임베딩을 새로고침합니다.
임베딩이 stale 또는 missing으로 보인다 Embedding Settings, Refresh Embeddings 임베딩 모델/키를 점검하고 현재 scope에서 다시 계산합니다.
설정을 여러 캐릭터에서 다르게 쓰고 싶다 Settings Profiles 캐릭터 성격에 맞는 프로필을 분리 저장하고 필요할 때 불러옵니다.
